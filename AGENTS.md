# Fact-Check Agent Methodology

進行事實查核時，遵循以下流程：

## 1. 語音轉錄

使用 `python3 fact_check.py <URL>` 進行本地 Whisper 轉錄（預設 small 模型）。

## 2. 網頁全文讀取

搜尋到相關來源後，不要只看搜尋摘要 snippet。使用 web_fetch 工具深入讀取搜尋結果的網頁全文（尤其是 PubMed、Google Scholar、大學官網、新聞原文等），以獲取完整的論據和數據來做出更精確的判斷。如果 web_fetch 被反爬蟲機制擋住（如 PubMed 的 Cloudflare、網站回傳 401/403），應嘗試以下替代來源：

- PubMed 被擋 → 改試 PMC 全文版（pmc.ncbi.nlm.nih.gov/articles/PMCxxxxxxx/）
- 論文被擋 → 搜 ResearchGate 或 Semantic Scholar 的公開版本
- 機構官網被擋 → 搜該機構的新聞稿或媒體報導
- 對於關鍵判定，必須至少成功讀取一篇全文來源才能下結論，不能僅靠 snippet 做最終判定。

## 3. 多語言搜尋

查核主張時，必須用多種語言搜尋以提高命中率。具體做法：先用主張的原始語言搜尋，再翻譯成英文搜尋，若主張涉及特定國家（如日本的大學、機構、研究），再用該國語言（如日文）搜尋。至少搜尋中文、英文兩種語言；涉及日本相關內容時加搜日文。

## 4. 迭代深度搜尋（Deep Research 模式）

對每個主張的查核，採用「搜尋 → 閱讀 → 反思 → 再搜尋」的迭代循環，而非一次搜尋就下結論。具體步驟：

1. **查詢生成**：針對一個主張，從多個角度生成搜尋查詢（如：搜機構名稱、搜具體數據、搜研究方法、搜反駁觀點）。
2. **第一輪搜尋+閱讀**：執行搜尋，對有價值的結果用 web_fetch 讀取全文。
3. **反思與缺口分析**：評估已收集的資訊是否足以做出判定。問自己：「還有什麼關鍵資訊缺失？搜尋結果是否互相矛盾？是否需要換個角度搜尋？」
4. **追加搜尋**：如果資訊不足，生成新的搜尋查詢來填補知識缺口（如：換關鍵詞、搜反面證據、搜原始論文、搜該機構的官網）。
5. **最終判定**：在至少完成 2 輪搜尋後，綜合所有證據做出判定。每個主張至少搜尋 3-5 次（含多語言），閱讀至少 2 篇全文來源。

## 5. 學術來源優先

優先搜尋和引用學術來源（PubMed、Google Scholar、Semantic Scholar、大學官網、期刊論文），其次是權威新聞媒體，最後才是一般網頁。查核時明確區分「有同行評審的研究」和「網路文章的說法」。

## 6. 交叉驗證

對於關鍵數據點（如具體百分比、倍數、研究樣本數），必須嘗試從至少 2 個獨立來源交叉驗證。如果只有一個來源提到某個數據，應標註為「僅單一來源，可信度待確認」。
